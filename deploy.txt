Great question. Right now without nginx your setup looks like:

```
internet ──► your server IP:8000 ──► fastapi
```

Problems with this:
- Port 8000 is ugly and non standard
- No SSL (HTTPS) — browsers will warn users
- Gunicorn/uvicorn shouldn't be exposed directly to the internet — not designed for it
- Can't serve multiple apps on same server

Nginx sits in front and fixes all of this:

```
internet ──► yourdomain.com (port 80/443) ──► nginx ──► fastapi:8000
```

---

### Updated docker-compose.yml

```yaml
services:
  redis:
    image: redis:alpine

  fastapi:
    build: .
    command: gunicorn main:app -w 4 -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000
    expose:
      - "8000"        # only expose to internal docker network, not to internet
    depends_on:
      - redis
    environment:
      - REDIS_URL=redis://redis:6379

  arq_worker:
    build: .
    command: arq worker.WorkerSettings
    depends_on:
      - redis
    environment:
      - REDIS_URL=redis://redis:6379

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"       # HTTP
      - "443:443"     # HTTPS
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf
      - ./certbot/conf:/etc/letsencrypt        # SSL certs
      - ./certbot/www:/var/www/certbot
    depends_on:
      - fastapi

  certbot:
    image: certbot/certbot
    volumes:
      - ./certbot/conf:/etc/letsencrypt
      - ./certbot/www:/var/www/certbot
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data  # data persists across restarts

volumes:
  postgres_data:

```

### nginx.conf

```nginx
server {
    listen 80;
    server_name yourdomain.com;

    # for certbot SSL verification
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # redirect all HTTP to HTTPS
    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl;
    server_name yourdomain.com;

    ssl_certificate /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;

    location / {
        proxy_pass http://fastapi:8000;   # "fastapi" = docker service name
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

Notice `proxy_pass http://fastapi:8000` — docker compose creates an internal network where services talk to each other by **service name**.

### Getting SSL cert (Let's Encrypt — free)

```bash
# 1. first start nginx without SSL just to get the cert
docker compose up -d nginx

# 2. get the cert
docker compose run certbot certonly --webroot \
  --webroot-path=/var/www/certbot \
  --email you@email.com \
  --agree-tos \
  --no-eff-email \
  -d yourdomain.com

# 3. now restart nginx with SSL config in place
docker compose restart nginx
```

### Linking your domain

On your domain registrar (Namecheap, GoDaddy, etc.) add an **A record**:

```
Type    Host    Value
A       @       your_server_ip
A       www     your_server_ip
```

Wait for DNS propagation (minutes to hours), then:

```bash
docker compose up -d
```

### Supabase Free Tier
```
PostgreSQL        ✅
500MB storage     ✅
Pauses after 1 week inactivity  ❌  (annoying for real apps)
2 free projects   ✅
Dedicated server  ✅ (not serverless)
```

### Neon Free Tier
```
PostgreSQL        ✅
512MB storage     ✅
Scales to zero    ← serverless, cold starts on first query
Branching         ✅ (nice for dev/staging)
Never pauses      ✅ (unlike supabase)
```

---

### The catch with both

```
Free tier is fine until:
  - Storage exceeds limit
  - You need more connections
  - Supabase pauses your project
  - Neon cold start latency is unacceptable

Then paid starts at:
  Supabase   $25/month
  Neon       $19/month
```

Both jump significantly from free to paid.

### Practical recommendation for your stack

```
Stage               DB choice
─────────────────────────────────────────────
Building/testing    Neon free tier
Early users         Postgres on same CX23
Growing traffic     Hetzner managed DB or Neon paid
Scale               AWS RDS
```

Neon free tier is genuinely the best starting point — no pausing, branching is useful for testing, and your CX23 at $4/month + Neon free = nearly free full stack. Only migrate when you actually hit limits.
